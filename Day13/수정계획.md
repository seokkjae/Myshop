# Day 13 — 수정 계획

## 배경 (왜 이 작업을 하는가)
- `Day05/index.html` 에 Supabase URL·**anon 키가 평문 하드코딩**되어 있고 이미 git 에 커밋된 상태였다.
- 예약 데이터를 **조회·수정·삭제할 관리자 화면이 없었다.**
- `reservations` 테이블은 RLS 가 켜져 있어 **anon 키로는 INSERT 만 허용**되고 SELECT/UPDATE/DELETE 는 차단됨(`Day11/create_reservations_table.sql`).
  → 관리 기능은 **반드시 service_role 키로 서버에서** 처리해야 하며, 정적 사이트에서 env 를 실제로 쓰려면 **서버리스 함수(`/api/*`)** 가 필요하다.

## 목표
1. `index.html` 에서 API 키 완전 제거 → **Vercel 환경변수**로 이전.
2. 비밀번호로 보호되는 **관리자 페이지 `Day13/admin.html`** 추가.
3. 신규 컬럼 **`status`(대기/확정/완료/취소)·`note`(메모)** 를 관리자 화면에서 운영.

## 결정 사항 (사용자 확인)
| 항목 | 선택 |
|------|------|
| 구조 | **Vercel 서버리스 함수** (`/api/*`, Node) — 키는 서버 env 에만 |
| admin 보호 | **비밀번호 게이트** — 서버가 `ADMIN_PASSWORD` 와 대조 후 service_role 작업 |
| status 값 | **대기 / 확정 / 완료 / 취소** (신규 = 대기) |

## 데이터 모델
기존: `id, name, phone, menu_preference, "time", count, created_at`
추가: **`status`(text), `note`(text)**

운영 SQL (`Day13/update_status_note.sql`):
- status 기본값 `'대기'` 지정 + 백필 + 허용값 CHECK 제약.

## 환경변수 (Vercel)
| 이름 | 용도 |
|------|------|
| `SUPABASE_URL` | 모든 함수 |
| `SUPABASE_ANON_KEY` | `/api/reserve` (공개 INSERT) |
| `SUPABASE_SERVICE_ROLE_KEY` | `/api/admin` 전용 (브라우저/깃 노출 금지) |
| `ADMIN_PASSWORD` | admin 게이트 대조 |

## 변경/생성 파일
- **(신규)** `api/reserve.js` — 공개 예약 접수(POST), anon 키, 서버측 재검증 후 `status='대기'` INSERT.
- **(신규)** `api/admin.js` — 비밀번호 인증 후 service_role 로 GET(목록)/PATCH(상태·메모)/DELETE.
- **(신규)** `Day13/admin.html` — 관리자 대시보드.
- **(수정)** `Day05/index.html` — Supabase CDN·하드코딩 키·`createClient` 제거 → `/api/reserve` fetch 호출(검증·토스트 유지).
- **(수정)** `.gitignore` — `.env`, `node_modules/`, `.vercel` 제외.
- **(신규)** `vercel.json` — `/admin` → `/Day13/admin.html` rewrite.
- **(신규)** `.env.example`, `Day13/update_status_note.sql`, `Day13/README.md`.

## admin.html 화면 설계 요약
- (A) 로그인 게이트(비밀번호 → `sessionStorage`) / 401 시 재로그인
- (B) 헤더(새로고침·로그아웃)
- (C) 통계 칩: 전체·대기·확정·완료·취소
- (D) 상태 필터 탭 + 이름·연락처 검색
- (E) 예약 카드: 이름·인원·연락처(tel)·메뉴·시간·접수시각 + **상태 배지/드롭다운**, **메모 인라인 편집**, 예약취소(소프트)/영구삭제
- (F) 인라인 PATCH(상태·메모), 실패 시 롤백 + 토스트
- 디자인: Day03 모노톤 톤·토큰 재사용, 모바일 우선.

## 보안 메모
- anon 키는 설계상 공개용이며 이미 git 히스토리에 노출됨 → RLS(INSERT 전용)가 보호선.
- **service_role 키는 절대 클라이언트/커밋 금지**, env 에만.
- 비밀번호는 HTTPS 로 매 요청 전송·서버 대조.

## 검증
1. SQL 실행 → `status`/`note` 확인.
2. `vercel env pull` → `vercel dev` (함수는 배포/`vercel dev`에서만 동작).
3. Day05 폼 제출 → `status='대기'` 행 생성 + 소스에 키 없음.
4. `/admin` 틀린 비번 401 / 정상 시 목록·통계.
5. 상태 변경·메모 저장·취소·삭제 동작 + Supabase 반영.
6. Vercel 배포 후 재확인.
