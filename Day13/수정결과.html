DAY 13 · 수정 결과 보고

관리자 페이지 + API 키 환경변수화

하드코딩된 Supabase 키를 서버 환경변수로 이전하고, 비밀번호로 보호되는 예약 관리자 화면을 추가했습니다.

1. 한눈에 보기

  • API 키 제거 — Day05/index.html 의 Supabase URL·anon 키·createClient 완전 삭제. 저장소 전체에서 키 값 사라짐.
  • 서버리스 함수 도입 — /api/reserve(공개 예약), /api/admin(관리). 키는 서버 환경변수에만 존재.
  • 관리자 페이지 신설 — Day13/admin.html (배포 시 /admin), 비밀번호 게이트로 보호.
  • status·note 운영 — 상태(대기/확정/완료/취소) 드롭다운 변경 + 관리자 메모 인라인 편집.

2. 파일별 변경 내역

구분파일내용
신규api/reserve.js공개 예약 접수(POST). anon 키로 PostgREST INSERT, 서버측 재검증, status='대기' 부여.
신규api/admin.js비밀번호(x-admin-password) 인증 후 service_role 로 GET·PATCH·DELETE (RLS 우회).
신규Day13/admin.html관리자 대시보드 (로그인·통계·필터·상태/메모 편집).
수정Day05/index.htmlSupabase CDN·하드코딩 키 제거 → /api/reserve fetch 호출. 검증·토스트 로직 유지.
수정.gitignore.env, node_modules/, .vercel 제외 추가.
신규vercel.json/admin → /Day13/admin.html rewrite.
신규.env.example필요한 환경변수 4종 예시.
신규Day13/update_status_note.sqlstatus 기본값·백필·CHECK 제약 SQL.
신규Day13/README.md설정·실행 절차.

3. 동작 흐름

[공개 예약] Day05 폼 ─POST /api/reserve─▶ (anon 키) ─▶ Supabase INSERT (status=대기)
[관 리] admin.html ─x-admin-password─▶ /api/admin ─(검증)─▶ (service_role) ─▶ Supabase
GET 목록 / PATCH 상태·메모 / DELETE

키는 모두 서버 함수의 process.env 에서만 읽으며, 브라우저로 전송되지 않습니다. 관리자 작업은 RLS 를 우회하는 service_role 키를 쓰되, 반드시 비밀번호 검증을 통과해야 합니다.

4. 관리자 화면 구성

영역내용
로그인 게이트비밀번호 입력 → sessionStorage 보관. 401 시 자동 재로그인.
통계 칩전체 · 대기 · 확정 · 완료 · 취소 건수 (클라이언트 집계).
필터 / 검색상태 탭 + 이름·연락처 부분일치 검색, 최신순 정렬.
예약 카드이름·인원·연락처(tel)·메뉴·시간·접수시각 + 색상 상태 배지.
상태 변경드롭다운 변경 → PATCH 즉시 반영, 배지·통계 갱신, 실패 시 롤백.
메모note 인라인 textarea + 저장 버튼.
취소 / 삭제기본은 status='취소'(소프트), 영구 삭제는 확인창 후 보조 동작.

5. 검증 상태

  • api/reserve.js, api/admin.js — node --check 문법 검사 통과.
  • 저장소 전체 grep — 하드코딩된 키 값(프로젝트 ID·키 문자열) 완전 제거 확인.
  • Day05/index.html — supabaseClient/createClient 잔존 없음 확인.

주의 · 서버리스 함수는 파일을 그냥 여는 방식으론 동작하지 않습니다. vercel dev 또는 배포 환경에서만 /api/* 가 응답합니다. 따라서 폼/관리자 실제 동작 확인은 Vercel 환경에서 수행해야 합니다.

6. 사용자가 할 일

  1. Supabase — Day13/update_status_note.sql 실행.
  2. Vercel 환경변수 등록(Production·Preview·Development): SUPABASE_URL, SUPABASE_ANON_KEY, SUPABASE_SERVICE_ROLE_KEY, ADMIN_PASSWORD.
  3. 로컬 확인 — vercel env pull → vercel dev.
  4. 배포 — vercel --prod 또는 git push 자동 배포.

잘 된 점 · service_role 키는 이번 작업의 어떤 코드·커밋에도 포함되지 않았습니다. 앞으로도 env 에만 보관하세요.

인지 필요 · 기존 anon 키는 이전 커밋(git 히스토리)에 남아 있습니다. 설계상 공개키이며 RLS(INSERT 전용)가 보호선이지만, 민감하다고 판단되면 Supabase 키 재발급을 고려하세요.
